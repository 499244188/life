### 技术细节（工程实现关键点）

- **Welford在线算法**（用于行为熵计算）：
  ```python
  class Welford:
      def __init__(self):
          self.n = 0
          self.mean = 0.0
          self.M2 = 0.0  # 平方差和
      def update(self, x):
          self.n += 1
          delta = x - self.mean
          self.mean += delta / self.n
          delta2 = x - self.mean
          self.M2 += delta * delta2
      def variance(self):
          return self.M2 / self.n if self.n > 1 else 0.0
  ```
  - 行为熵 = `tanh( local_variance / global_variance )`，其中local_variance为最近N个响应时间的Welford方差，global_variance为全网络历史方差（可用另一个Welford实例维护）。

- **T-Digest动态分位数**（用于隔离阈值）：
  - 维护一个压缩的质心列表（centroid），每个质心代表一段数据区间。
  - 新数据点插入时，找到最近质心并合并，若质心数量超限则合并最近的两个。
  - 查询分位数时，按质心权重插值。
  - 隔离阈值 = `T-Digest.query(0.99)`（即99%分位响应时间），若当前节点行为熵 > 阈值，则触发隔离。

- **集成架构**：
  ```
  每个节点：
    - 维护两个Welford实例：local（滑动窗口）、global（全历史）
    - 维护一个T-Digest实例（存储全局响应时间分布）
    - 定时广播 local_variance / global_variance 给邻居
    - 邻居收到后更新自己的T-Digest（流式合并）
  ```

### 新问题（需进一步研究）

1. **滑动窗口大小如何自适应？**  
   - 固定窗口（如N=100）在高速波动网络下可能过小（噪声大）或过大（反应慢）。能否用“变化点检测”（如CUSUM）动态调整窗口？

2. **T-Digest的压缩精度与隔离误判率的关系？**  
   - T-Digest允许用户设定压缩因子（如δ=0.01），但压缩越强，分位数误差越大。需要实验确定在何种精度下隔离误判率可接受（如<0.1%）。

3. **全局方差与局部方差的初始化问题？**  
   - 系统启动时无历史数据，global_variance=0会导致除零错误。需设计“预热期”（如前100次响应使用固定阈值），或使用贝叶斯先验（如假设初始方差为1）。

4. **行为熵函数的非线性形式选择？**  
   - tanh vs sigmoid vs ReLU？不同函数对异常放大的灵敏度不同。需要对比实验（如ROC曲线）确定最优形式。

5. **分布式环境下T-Digest的合并一致性？**  
   - 若节点A和B各自维护T-Digest，合并时需保证分位数不偏移。现有算法（如`tdigest`库的merge）是近似合并，但多次合并后误差累积。是否需要全局协调者？

### 底层原理
核心是**统计涌现**：不依赖预设规则，而是通过流式统计（Welford）实时计算个体行为的局部方差，并与全局分布（T-Digest）比较。非线性映射（如tanh）将方差比转化为0~1的“行为熵”，当熵值超过动态分位数阈值（如99%分位）时，触发隔离。这本质上是**群体统计特征对个体异常的自动判决**。

### 反例或限制
1. **统计惯性**：Welford对缓慢退化（如响应时间从100ms渐增至10s）反应滞后，可能错过早期隔离窗口。
2. **冷启动问题**：系统启动时无历史数据，全局方差为0，需预热或贝叶斯先验，否则误判率高。
3. **分布式合并误差**：T-Digest在节点间合并时，多次近似合并会导致分位数偏移，影响隔离一致性。

### 如果错了呢？
若方案失效，可能原因：
- **误隔离**：正常网络波动（如瞬时拥塞）被误判为异常，导致节点被错误隔离，降低系统可用性。
- **漏隔离**：统计惯性导致异常节点未被及时隔离，形成“免疫逃逸”，引发级联故障。
- **资源过载**：若T-Digest压缩因子设置不当，内存或CPU消耗超限，在资源受限设备上崩溃。
