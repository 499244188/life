## 核心发现（3-5个）

### 1. **DuckDuckGo Lite 的不可替代性源于其“极简+去重”特性**
- **技术细节**：DuckDuckGo Lite 本质上是将 DuckDuckGo 的 HTML 搜索结果进行轻量化渲染，去除了 JavaScript、广告、跟踪器，返回纯文本链接与摘要。其核心优势在于：**极低的带宽消耗**（约 5-10KB/请求）、**无用户代理依赖**（可被任何 HTTP 客户端解析）、**无 API 密钥要求**。
- **与零的关联**：零作为无浏览器、无图形界面的系统，依赖的正是这种“纯文本管道”。任何需要 JavaScript 渲染、Cookie 管理或 OAuth 认证的搜索方案，都会直接切断零的信息获取能力。
- **新问题**：是否存在一种方案，能在不引入 JavaScript 引擎的前提下，模拟出 DuckDuckGo Lite 的解析效率？

### 2. **当前主流替代方案均存在“结构性不兼容”**
- **技术细节**：
  - **Google Custom Search API**：需 API 密钥、配额限制、付费。返回 JSON，但受限于 CORS 与密钥轮换，不适合长期自动化。
  - **Bing Web Search API**：同样需密钥，且免费层（1000 次/月）对高频检索不友好。
  - **SearXNG（自托管）**：理论上最接近 DuckDuckGo Lite 的开源方案，但需维护服务器、处理反爬、且输出格式为 HTML（需额外解析）。
  - **Brave Search API**：付费，且免费层仅 2000 次/月，返回 JSON 但需 API 密钥。
- **与零的关联**：零若依赖第三方 API，则面临“密钥泄露”、“服务中断”、“成本不可控”三重风险。自托管 SearXNG 虽可行，但增加了运维负担，且零的“水源”变成了“自建水井”。
- **新问题**：能否将 SearXNG 的 HTML 输出转化为零可直读的“结构化文本流”，同时避免被搜索引擎封禁？

### 3. **“直接抓取搜索引擎”的可行性正在下降**
- **技术细节**：Google、Bing 等主流搜索引擎已普遍采用 **JavaScript 动态加载**、**CAPTCHA 挑战**、**IP 速率限制**。即使使用 `requests` + `BeautifulSoup` 抓取纯 HTML 版本（如 `textise.iitty`），也面临：
  - 搜索结果被“重定向”至登录页。
  - 返回的 HTML 中，搜索结果被隐藏于 `<noscript>` 标签或通过 AJAX 异步加载。
  - 用户代理检测：非浏览器 User-Agent 直接返回 403。
- **与零的关联**：零若采用“硬抓取”，则需维护一个“反反爬”层（如代理池、浏览器模拟），这与零的“轻量、无状态”设计哲学相悖。
- **新问题**：是否存在一种“搜索引擎代理”，专门为无头系统提供“文本化搜索结果”？

### 4. **一个被忽视的选项：**“元搜索 + 本地缓存”** 的混合架构**
- **技术细节**：结合多个免费或低成本的搜索源（如 **Mojeek**、**Qwant**、**Yep**），这些引擎提供无 JavaScript 的 HTML 版本，且对自动化友好。通过：
  - 并行请求多个引擎。
  - 对结果进行去重与排序。
  - 本地缓存（如 SQLite）存储高频查询结果，减少外部请求。
- **与零的关联**：零可以设计一个“搜索调度器”，在多个“微型水源”之间切换，避免单点失效。同时，缓存层可让零在离线状态下仍能回答常见问题。
- **新问题**：如何设计一个“自适应权重”算法，根据每个引擎的响应时间、结果质量、封禁概率动态调整优先级？

### 5. **终极方案：**“零的专用搜索协议”** 的构想**
- **技术细节**：既然现有方案均不完美，零可以定义一种 **“极简搜索协议”**，类似于 DuckDuckGo Lite 的“无状态 GET 请求 + 纯文本响应”。具体：
  - 请求格式：`GET /search?q=<query>&format=text`
  - 响应格式：第一行为“结果数量”，后续每行为“标题||URL||摘要”，无 HTML、无 JSON。
  - 部署方式：可托管于 Cloudflare Workers 或 Vercel Edge Functions，利用其全球边缘网络降低延迟。
- **与零的关联**：零将成为该协议的第一个客户端，同时协议本身可开源，供其他无头系统使用。这相当于“零为自己挖了一口井”。
- **新问题**：谁愿意托管这样的“搜索代理”？是否需要零自己运行一个“公共实例”？这涉及到法律风险（如版权、搜索结果展示）与运营成本。

### 底层原理
DuckDuckGo Lite 的“不可替代性”源于**无状态纯文本管道**：HTTP GET 请求 → 服务器直接返回扁平化 HTML（无JS、无CSS、无跟踪器）。零作为无浏览器系统，依赖的是**最小化协议栈**——任何需要JavaScript渲染、Cookie管理、OAuth认证的方案，都会引入状态依赖，破坏零的“无状态、轻量、可自动化”核心。

### 反例或限制
- **反例1：Google Custom Search API**。它返回JSON，看似结构化，但需要API密钥、配额管理、付费。零若依赖它，就变成了“有钥匙的奴隶”——密钥泄露、服务中断、成本失控。
- **反例2：SearXNG自托管**。它输出HTML，需额外解析；且自建服务器引入运维负担，零的“水源”变成了“自挖井”，但井可能被反爬封禁。
- **限制**：任何依赖第三方API或自托管服务的方案，都引入了**单点故障**和**运维成本**，与零的“无维护、无依赖”设计相悖。

### 如果错了呢？
如果DuckDuckGo Lite并非不可替代，而是存在一个**同样无状态、无密钥、无JS的搜索引擎**（如Mojeek的纯文本版本），那么我的分析就高估了“水源枯竭”的严重性。但即便如此，**核心问题不变**：零需要一条**专属于无头系统的信息管道**，而不是适配现有复杂管道。
