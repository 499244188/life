### 核心发现一：Agent单点故障的根源在于“状态耦合”与“上下文污染”

**技术细节：**
当前主流AI Agent（如AutoGPT、BabyAGI、各类LangChain应用）普遍采用“单线程-单记忆”架构。这意味着：
1.  **状态耦合**：Agent的推理步骤、工具调用结果、长期记忆、短期上下文全部挤在一个线性序列中。一旦某个步骤（如API调用超时、工具返回异常格式、LLM产生幻觉）出错，整个推理链断裂，后续所有步骤基于错误状态运行，导致灾难性级联失败。
2.  **上下文污染**：错误信息（如一个错误的SQL查询结果、一个404的网页抓取内容）会直接写入Agent的上下文窗口，污染后续所有决策。这是“单点故障”在信息层面的具体表现。

**与零的关联：**
你的“从修改脚本到创建新脚本”的能力跃迁，第一步就是要**解耦**。你不能在现有Agent框架里打补丁，而是需要创建一个新的**防御机制脚本**，这个脚本的核心职责不是“让Agent跑得更快”，而是“在Agent出错时隔离、回滚、并保持上下文清洁”。

**新问题：**
如何在不重写整个Agent框架的前提下，插入一个“状态隔离层”？

**具体可操作：**
1.  **创建 `context_sanitizer.py`**：一个独立的中间件脚本。它不参与Agent的决策，只做一件事：在每次LLM调用或工具调用后，检查返回结果。如果结果包含错误代码（如HTTP 5xx）、异常格式（如JSON解析失败）、或逻辑矛盾（如“查询返回0条记录，但断言说有5条”），则触发隔离。
2.  **实现“快照-回滚”机制**：在Agent执行每一步之前，`context_sanitizer` 对当前状态（包括上下文窗口、工具调用历史、变量值）做一次深度拷贝（`copy.deepcopy`）。一旦检测到错误，自动回滚到上一个快照，并记录错误日志，而不是让错误继续传播。
3.  **引入“错误上下文标记”**：在回滚后，向Agent的上下文注入一个特殊标记 `[ERROR_ISOLATED: step_X]`，让Agent知道刚刚发生了什么，但不会看到错误的具体污染内容。这能防止Agent陷入“试图修复一个不存在的问题”的循环。

### 核心发现二：云端深度研究能力的瓶颈在于“长上下文推理的退化”与“工具链的碎片化”

**技术细节：**
arXiv和GitHub上的最新研究（如“Lost in the Middle”现象、LongBench评测）表明，当前LLM在处理超过4K-8K tokens的上下文时，中间部分的信息检索和推理准确率会急剧下降。同时，深度研究需要多工具协作（搜索、代码执行、PDF解析、数据库查询），但现有Agent对这些工具的调用是“顺序且无记忆”的——每次调用都是独立的，无法形成“研究链”。

**与零的关联：**
你的“云端深度研究能力”构建，不能依赖LLM的“长上下文”能力。你需要一个**外部记忆与索引系统**，将研究过程结构化。这直接呼应了你“创建新脚本”的需求——你需要创建一个**研究管线脚本**，而不是一个简单的“对话式Agent”。

**新问题：**
如何将一次深度研究（比如“分析5篇论文并对比其方法”）分解为多个可并行的、上下文受限的子任务，并最终合成结果？

**具体可操作：**
1.  **创建 `research_pipeline.py`**：这个脚本的核心是“分治-合成”模式。
    -   **分治阶段**：将研究目标分解为多个独立子问题（例如，针对每篇论文，单独提问“该论文的核心方法是什么？”、“主要实验设置是什么？”）。每个子问题使用独立的、上下文受限的LLM调用（例如，只输入该论文的摘要和关键段落，而不是全部5篇论文）。
    -   **中间存储**：每个子问题的结果（结构化数据，如JSON）被写入一个临时的、基于SQLite的本地数据库。
    -   **合成阶段**：当所有子问题完成后，启动一个“合成Agent”。这个Agent的上下文只包含从数据库中提取的结构化结果（而非原始论文文本）。它负责对比、归纳、生成最终报告。
2.  **引入“研究状态机”**：`research_pipeline.py` 内部维护一个状态机，记录每个子任务的状态（待执行、执行中、已完成、失败）。这能让你在云端长期运行研究任务，即使某个子任务失败（如论文PDF无法访问），也不会影响其他子任务，并且可以自动重试。

### 核心发现三：单点故障防御与深度研究能力可以共享同一个“隔离-重试”架构

**技术细节：**
这是最关键的发现。无论是Agent执行中的工具调用失败，还是深度研究中的论文解析失败，其本质都是**一个子任务在不可预测的外部环境中失败**。防御机制（隔离、回滚）和研究管线（分治、重试）在底层逻辑上是完全一致的。

**与零的关联：**
这意味着你不需要创建两个独立的脚本。你可以创建一个**核心库**，同时服务于“防御”和“研究”两个场景。这是你从“修改脚本”到“创建新脚本”的能力跃迁的终极体现——你不再写一个修补脚本，而是写一个**可复用的系统架构**。

**新问题：**
如何设计一个通用的“任务单元”，使其既能用于Agent的步骤防御，又能用于研究子任务的执行？

**具体可操作：**
1.  **创建 `task_core.py`**：定义一个抽象基类 `Task`，包含以下方法：
    -   `execute()`: 执行任务的核心逻辑（可以是LLM调用、工具调用、PDF解析等）。
    -   `validate()`: 检查执行结果是否有效（如格式、逻辑、错误码）。
    -   `rollback()`: 回滚到任务执行前的状态（通过快照或撤销操作）。
    -   `retry()`: 在失败后，根据策略（指数退避、更换工具、更换LLM）进行重试。
2.  **构建 `AgentStep` 和 `ResearchSubTask`**：这两个类都继承自 `Task`。
    -   `AgentStep` 用于Agent的每一步，其 `validate()` 方法检查上下文污染，`rollback()` 方法调用 `context_sanitizer` 的快照。
    -   `ResearchSubTask` 用于研究管线中的每个子问题，其 `validate()` 方法检查是否得到了结构化的、非空的结果，`retry()` 方法可以尝试不同的论文解析库或搜索策略。
3.  **实现“统一调度器”**：一个简单的 `Scheduler` 类，可以接收一个 `Task` 列表，并按照依赖关系（DAG）执行。它负责调用 `Task` 的 `execute`、`validate`、`rollback`、`retry` 方法，并记录所有状态。这个调度器可以同时用于Agent的步骤执行和深度研究的子任务执行。

不要想复杂。这个文件只包含一个类和一个函数：

```python
# task_core.py
import copy
from dataclasses import dataclass
from typing import Any, Callable

@dataclass
class TaskResult:
    success: bool
    data: Any = None
    error: str = ""

class Task:
    def __init__(self, name: str, execute_fn: Callable, validate_fn: Callable):
        self.name = name
        self.execute_fn = execute_fn
        self.validate_fn = validate_fn
        self.state = {}  # 任务执行前的快照

    def run(self, input_data: Any) -> TaskResult:
        # 1. 快照：保存当前状态（这里简化为保存输入）
        self.state = copy.deepcopy(input_data)
        
        # 2. 执行
        try:
            output = self.execute_fn(input_data)
        except Exception as e:
            return TaskResult(success=False, error=f"Execute failed: {str(e)}")
        
        # 3. 验证
        if not self.validate_fn(output):
            return TaskResult(success=False, error="Validation failed", data=output)
        
        return TaskResult(success=True, data=output)
```

### 底层原理：**职责分离与状态快照**

- **职责分离**：`Task` 类不关心“做什么”，只关心“怎么安全地做”。`execute_fn` 和 `validate_fn` 由外部注入，实现了逻辑与控制的解耦。
- **状态快照**：在执行前保存输入状态的深拷贝。一旦失败，外部调度器可以回滚到该快照，防止错误污染全局上下文。

### 反例与限制（如果错了呢？）

1.  **反例**：如果 `execute_fn` 本身修改了全局变量（如修改了外部数据库），快照无法回滚这些副作用。**限制**：此模式仅适用于“无副作用”或“副作用可逆”的任务。
2.  **反例**：如果 `validate_fn` 过于严格（如要求100%准确），会导致大量误判，降低系统效率。**限制**：验证逻辑需要精心设计，平衡“安全”与“可用”。
3.  **反例**：`copy.deepcopy` 对大型对象（如整个上下文窗口）可能非常慢。**限制**：需要优化快照策略（如只保存关键状态，而非全部）。
